# API Overview

The AIO Sandbox API exposes a unified runtime for agents and developer tooling through a single HTTP surface. Instead of splitting browser, shell, files, notebooks, and MCP tools into separate services, the platform presents them as one versioned API.

The current OpenAPI contract is available at `/v1/openapi.json`, and the interactive reference is exposed in the docs portal at `/en/api`.

## Base Information

- OpenAPI version: `3.1.0`
- Documented service version: `1.9.4`
- Default local base URL: `http://127.0.0.1:8080`

Primary runtime entry points:

- REST API: `/v1/*`
- MCP Streamable HTTP: `/mcp` and `/v1/mcp`
- Browser CDP endpoint: `/cdp/json/version`
- Jupyter UI: `/jupyter`

## Authentication

The API supports two main authentication flows:

1. Short-lived ticket flow
- `POST /tickets` creates a temporary ticket.
- `GET /auth` validates a request using that ticket.

2. JWT flow
- `GET /auth` also accepts a JWT in the `Authorization` header.

This design fits edge-proxy and `auth_request` patterns, where a reverse proxy delegates auth decisions to the sandbox runtime.

## Response Pattern

Most endpoints return either a generic `Response` wrapper or a more specialized response model. The OpenAPI spec also documents compatibility layers in some domains, especially browser actions, where newer unified response fields coexist with older fields.

For automation clients, this means:

- treat `status` as lifecycle state, not always success or failure;
- check `exit_code` for command outcomes where relevant;
- prefer documented response models over assumptions based on one endpoint family.

## Typical Agent Workflows

### Read, edit, and validate code

1. Read files with `/v1/file/read`
2. Search with `/v1/file/search`, `/v1/file/grep`, or `/v1/file/glob`
3. Update with `/v1/file/write` or `/v1/file/replace`
4. Run verification through `/v1/bash/exec` or `/v1/shell/exec`

### Browser-driven task execution

1. Navigate with `/v1/browser/page/navigate`
2. Interact with `/v1/browser/page/click`, `/v1/browser/page/fill`, or `/v1/browser/page/hot_key`
3. Inspect content with `/v1/browser/page/text`, `/v1/browser/page/html`, or `/v1/browser/page/markdown`
4. Export evidence using screenshots, recordings, or HAR

### MCP tool orchestration

1. Enumerate servers with `/v1/mcp/servers`
2. Enumerate tools with `/v1/mcp/{server_name}/tools`
3. Execute tools with `/v1/mcp/{server_name}/tools/{tool_name}`

## Recommended Usage Notes

- Use `shell` when you want richer terminal-oriented workflows and terminal URLs.
- Use `bash` when you want process-oriented execution with explicit polling semantics.
- Use `browser/page/*` for deterministic automation and `browser/actions` for higher-level abstractions.
- Use MCP when your orchestration layer already speaks tool-calling natively.
- Use file watch endpoints for event-driven agent workflows instead of manual polling loops where possible.
